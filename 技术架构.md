## 2. 技术调研与借鉴结论

### 2.1 直接借鉴 Graphify 的设计思想

| 设计点 | 借鉴方式 | 说明 |
|--------|----------|------|
| 分阶段 Pipeline | 直接借鉴 | 降低复杂度与成本 |
| 置信度标签 | 直接借鉴 | 区分事实、推断和待审内容 |
| 增量缓存 | 直接借鉴 | 避免重复处理 |
| Hook 注入 | 重点借鉴 | 用机制保证命中经验，而不是靠 AI 自觉 |
| 图结构分析 | 部分借鉴 | 后期用于热点与路径分析 |

### 2.2 不直接照搬的部分

| Graphify 能力 | EKG 处理方式 | 原因 |
|---------------|--------------|------|
| AST / Tree-sitter 强结构解析 | 改为规则解析 + 手动输入 | 经验数据没有天然 AST |
| 代码依赖边 | 改为经验、概念、技术之间的语义边 | 对象不同 |
| 立刻启用社区发现 | 延后到 Phase 2/3 | 小样本经验图聚类不稳定 |
| 先做复杂可视化 | 延后 | 首版重点不是图美观，而是经验可用 |

## 3. 方案对比

### 方案一：本地文件化 + JSON 主索引 + 可选 NetworkX

- 技术栈：
  - 文件系统目录
  - JSON 主索引
  - Markdown 经验文件
  - 可选 Python/Node 脚本
  - 可选 NetworkX 用于后续图分析
- 优点：
  - 上手快
  - 迁移成本低
  - 与 skill 体系天然契合
  - 适合 Phase 1 快速验证
- 缺点：
  - 图查询能力初期较弱
  - 并发写入与多人共享能力有限
- 月成本：
  - 近似 0

### 方案二：Neo4j / 图数据库服务化

- 技术栈：
  - Neo4j
  - 持久服务
  - 结构化查询 API
- 优点：
  - 图查询强
  - 可扩展性好
  - 后续适合 MCP 化
- 缺点：
  - 首版过重
  - 运维复杂
  - 与“可复制、可移植”的首版目标冲突
- 月成本：
  - 中

### 方案三：向量检索优先

- 技术栈：
  - embedding
  - 向量库
  - LLM 语义检索
- 优点：
  - 语义搜索自然
  - 可处理模糊查询
- 缺点：
  - 成本高
  - 首版可解释性差
  - 容易出现“像是命中但说不清为什么”
- 月成本：
  - 中到高

### 对比表

| 维度 | 方案一 | 方案二 | 方案三 |
|------|--------|--------|--------|
| 启动速度 | 高 | 低 | 中 |
| 实现复杂度 | 低 | 高 | 中 |
| 可迁移性 | 高 | 中 | 中 |
| 首版可解释性 | 高 | 高 | 低 |
| 成本 | 低 | 中 | 中高 |

**推荐方案**：方案一

## 4. 推荐方案与改造策略

- 推荐方案：本地文件化 + JSON 主索引 + 轻量 Hook + 后续可升级图分析
- 为什么不是服务化重做：
  - EKG 首版验证的是“经验是否有价值”，不是“图数据库是否强大”
  - 首版要适配 skill 目录本地化使用场景
- 直接复用部分：
  - 当前 `SKILL.md`
  - 当前 skill 体系中的文档标准、任务拆解方式
- 必须新增部分：
  - `ekg.json`
  - `state.json`
  - `experiences/`
  - `reports/EKG_REPORT.md`
  - `hooks/`

## 5. 系统架构

```text
输入层
  ├── 手动输入（/ekg add）
  ├── 查询输入（/ekg query / explain / path）
  └── 后续自动输入（git / 对话 / 注释）

处理层
  ├── Pass 1: 规则解析
  ├── Pass 2: LLM 提取（Phase 2）
  ├── Build: 写入节点与关系
  ├── Analyze: 统计热点与失效状态
  └── Query: 检索与排序

存储层
  ├── experiences/*.md
  ├── ekg.json
  ├── state.json
  └── reports/EKG_REPORT.md

交互层
  ├── /ekg 命令族
  ├── PreToolUse Hook
  └── 后续 MCP / 可视化
```

## 6. Pipeline 设计

### Phase 1 Pipeline

```text
手动输入
  -> 结构化 Experience 条目
  -> 写入 experiences/ 与 ekg.json
  -> 更新 state.json
  -> 刷新 EKG_REPORT.md
```

### Phase 2 Pipeline

```text
规则检测（git / 注释 / commit）
  -> 判断是否值得提取
  -> LLM 生成候选经验
  -> 写入 UNCERTAIN / INFERRED
  -> review 后提升为 CONFIRMED
```

### Phase 3 Pipeline

```text
经验规模达到阈值
  -> 图构建
  -> 聚类分析
  -> 路径查询
  -> MCP / 可视化输出
```

## 8. 核心对象设计

### 8.1 节点类型

| 节点类型 | 说明 | 首版是否必须 |
|----------|------|--------------|
| Experience | 经验主体节点 | 是 |
| Concept | 领域概念 | 是 |
| Tech | 技术对象 | 是 |
| Project | 项目作用域 | 建议 |

### 8.2 关系类型

| 关系 | 含义 | 首版是否必须 |
|------|------|--------------|
| involves | 经验涉及某概念/技术 | 是 |
| relates_to | 经验与经验相关 | 建议 |
| solves | 某经验解决某问题模式 | 建议 |
| applies_to | 经验适用于某项目 | 建议 |
| caused_by | 经验根因关联 | Phase 2 更有价值 |

## 9. 经验状态与置信度

### 9.1 置信度枚举

| 枚举 | 含义 | 用途 |
|------|------|------|
| CONFIRMED | 用户或人工已确认 | 优先注入 |
| INFERRED | 系统根据上下文推断 | 可展示但不优先 |
| UNCERTAIN | 不确定、待审核 | 默认不主动注入详细内容 |

### 9.2 生命周期状态

| 枚举 | 含义 |
|------|------|
| ACTIVE | 当前有效 |
| NEEDS_REVIEW | 文件变化较大或长期未验证 |
| STALE | 大概率过期 |
| ARCHIVED | 归档，不再默认参与检索 |

## 10. Hook 设计

### 10.1 分级注入

| 级别 | 行为 | 默认策略 |
|------|------|----------|
| Level 0 | 静默 | 无命中或同会话已提醒 |
| Level 1 | 一句话摘要 | 默认 |
| Level 2 | 完整经验摘要 | 用户主动查询或高置信度高匹配 |

### 10.2 触发策略

- 同一会话、同一文件默认只注入一次
- 只在编辑前或高风险查询前触发
- 不命中时不输出噪音

## 11. 增量与缓存策略

### 11.1 Phase 1

- 使用 `state.json` 记录：
  - 经验总数
  - 上次刷新报告时间
  - 待审经验数量
- 不做复杂 SHA256 级别缓存

### 11.2 Phase 2

- 对锚定文件记录 hash 或 git commit
- 自动提取仅处理上次状态之后的新输入

## 12. 技术风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 自动提取噪声过高 | 污染经验库 | Phase 1 不自动提取；Phase 2 默认 `UNCERTAIN` |
| Hook 打扰编码 | 使用体验差 | 默认 Level 1 摘要；同会话去重 |
| 经验快速过期 | 误导 AI | 引入 `NEEDS_REVIEW / STALE` |
| 冷启动无命中 | 用户感知无价值 | 预置 L3 种子经验 |
| 图分析过早上马 | 投入大、收益小 | 经验数不足时不启用聚类 |
