# EKG 技术架构

## 1. 架构定位

EKG 是一个面向编码代理与工程协作的本地知识图谱系统。

它解决的不是“替代代码搜索”，而是把已经验证过的工程经验、研究论文、项目上下文和复盘候选沉淀为可查询、可解释、可回放的本地记忆层。

当前版本的架构目标有四个：

1. 在改代码前尽可能回忆已有经验，而不是重复踩坑。
2. 在任务结束后先生成可审查候选，而不是直接写入正式知识。
3. 把经验知识与研究论文知识放进同一张图里，支持联查与解释。
4. 保持本地优先、可迁移、可审计，不依赖中心化服务。

## 2. 当前阶段判断

当前项目适合描述为 `Phase 2+ / Phase 3 preview`：

- Phase 1 已完成：本地知识记录、查询、解释、基础 Hook、报告可用。
- Phase 2 大部分已完成：capture/review、SQLite 主存储、项目上下文、多 agent 写锁、备份恢复、宿主安装、stale 检测原型已落地。
- Phase 3 预览已可用：图路径、静态/服务端 panel、图视图、论文层与多语言概念检索已接入。

仍未完成的方向主要是：

- 更深入的自动 ingest 来源
- 更强的 stale/演化治理
- MCP / query server
- 更系统的图分析与聚类

## 3. 核心架构原则

### 3.1 本地优先

- 数据默认保存在本机。
- CLI、Hook、Panel 都围绕本地运行时工作。
- 即使导入外部论文元数据，导入后的正式数据仍落地到本地存储。

### 3.2 SQLite 主存储，JSON/Markdown 兼容导出

- 正式运行时以 `ekg-out/ekg.sqlite` 为主。
- `ekg-out/ekg.json`、`ekg-out/state.json`、`ekg-out/reports/EKG_REPORT.md` 作为兼容导出。
- `experiences/*.md` 与 `papers/*.md` 作为可阅读副本。

这样做的原因是：

- SQLite 更适合并发写入、增量迁移与结构化存储。
- JSON/Markdown 仍然适合审查、diff、迁移和备份。

### 3.3 可解释优先

- 查询并非黑盒 embedding 检索。
- 当前检索采用“规则 + 轻量语义”的混合模式。
- 每次命中都会返回理由，如 token 命中、topic 命中、canonical concept 命中等。

### 3.4 先候选，后确认

- 自动化流程不直接写入 CONFIRMED 经验。
- Stop / task-complete / ingest 首先创建 capture candidate。
- 只有经过 review/accept 后，候选才会进入正式知识图谱。

## 4. 分层架构

当前实现可以分成六层。

### 4.1 交互层

对外入口包括：

- CLI：`node scripts/ekg.js ...`
- Claude Hook：`PreToolUse`、`Stop`、`SubagentStop`
- Codex Hook：`SessionStart`、`UserPromptSubmit`、`PreToolUse/PermissionRequest(Bash)`、`Stop`
- 本地 Panel：静态导出或内置 HTTP 服务

### 4.2 运行时层

运行时由 `buildRuntime/loadRuntime/mutateRuntime/saveRuntime` 负责，核心职责是：

- 统一加载 config、index、state、storage backend
- 在修改前后刷新索引和统计
- 通过写锁保护多 agent 并发写入

### 4.3 领域模型层

当前一等实体主要有：

- Experience：工程经验
- Paper：研究论文
- Project：项目上下文
- Capture Candidate：待审查候选

图中还会动态生成虚拟节点或辅助节点，例如：

- tag
- tech
- file
- concept
- topic
- keyword
- author
- venue
- canonical

### 4.4 检索与图谱层

核心能力包括：

- `queryExperiences`
- `queryPapers`
- `traceExperiences`
- `buildKnowledgeGraph`
- `explain`
- `path`
- `survey`

这一层既支持直接检索，也支持在图中找邻接关系和路径。

### 4.5 持久化层

持久化实现包括：

- SQLite backend
- JSON backend
- Markdown 副本写出
- portable backup export/import

### 4.6 宿主集成层

宿主集成负责把 EKG 的工作流嵌入代理环境，包括：

- 安装 Hook
- 生成 Codex 指令文件
- 生成全局 AGENTS override
- 在会话开始、用户提问、执行 Bash、Stop 时注入提示或保护

## 5. 数据流

### 5.1 查询流

典型流程：

1. 用户或 Hook 提供关键词、文件路径或任务描述。
2. 运行时加载 `index + state + config`。
3. 检索层先做规则命中，再叠加轻量语义评分。
4. 若需要解释，图谱层补充邻接节点、关系和路径。
5. CLI/Hook/Panel 返回结果。

### 5.2 写入流

典型流程：

1. 通过 CLI 或 Hook 发起写操作。
2. 获取 `.ekg.lock` 写锁。
3. 修改运行时对象。
4. 刷新统计、索引、图谱相关导出。
5. 写入 SQLite。
6. 导出 JSON/Report/Markdown 镜像。

### 5.3 捕获流

自动或半自动 capture 流程：

1. `task-complete` / `ingest` 收到任务摘要或变更线索。
2. 生成 capture candidate。
3. 进入 `NEEDS_REVIEW / UNCERTAIN`。
4. 人工执行 `capture-accept` 或 `capture-dismiss`。
5. 接受后转为正式 Experience。

### 5.4 论文导入流

论文层流程：

1. `paper-import` 调 OpenAlex 或 Semantic Scholar。
2. 把返回结果标准化成 `Paper` 模型。
3. 自动补齐 aliases、canonical_terms、suggested_canonical_terms。
4. 与现有论文按 DOI / arXiv / title 去重合并。
5. 写入 SQLite、JSON、Markdown。

## 6. 存储架构

## 6.1 当前主存储

- 主存储：`ekg-out/ekg.sqlite`
- 导出索引：`ekg-out/ekg.json`
- 导出状态：`ekg-out/state.json`
- 导出报告：`ekg-out/reports/EKG_REPORT.md`
- Markdown 副本：`experiences/`、`papers/`

## 6.2 SQLite 模型

主要表：

- `experiences`
- `papers`
- `edges`
- `documents`

其中：

- `experiences` 和 `papers` 保存实体主体
- `edges` 保存显式边
- `documents` 保存 `index_meta`、`state` 等文档对象

## 6.3 为什么不直接只用 JSON

旧版本更偏向纯 JSON。

当前改为 SQLite 主存储，原因是：

- 更适合多 agent 写入
- 更容易做 schema 迁移和增量列扩展
- 能保留 JSON 镜像而不失去可读性

## 7. 查询与排序架构

## 7.1 Experience 检索

经验检索包含两类信号：

- 规则信号：tag、tech、file、concept、文本 token
- 语义信号：别名展开、trigram overlap、field coverage、canonical concept overlap

最终分数 = lexical score + semantic boost。

## 7.2 Paper 检索

论文检索与经验检索结构类似，但字段换成：

- title
- abstract
- summary
- findings / limitations / notes
- authors
- topics
- keywords
- venue / year
- canonical_terms

## 7.3 多语言 canonical concept

当前配置支持：

- alias 扩展
- multilingual concept registry
- exact canonical hit
- suggested canonical terms

这使得中英混合输入可以命中同一概念，例如：

- 登录跳转
- signin redirect
- auth redirect

## 8. 图谱架构

图谱不是单独存储在图数据库里，而是由运行时索引动态构建。

### 8.1 节点来源

- Experience
- Paper
- tag / tech / file / concept
- topic / keyword / author / venue / canonical

### 8.2 边来源

- Experience / Paper 自身的 relations
- tag / tech / file / concept 的归属关系
- topic / keyword / author / venue / canonical 的归属关系

### 8.3 图谱用途

- `explain`
- `path`
- `trace`
- Panel Graph View
- 经验与论文互相挂接

## 9. Hook 架构

## 9.1 Claude

Claude 集成目前主要挂在：

- `PreToolUse`
- `Stop`
- `SubagentStop`

职责分别是：

- 编辑前提醒
- 任务结束捕获
- 子代理结束捕获

## 9.2 Codex

Codex 集成支持三种模式：

- `light`
- `hooks`
- `strong`

其中 `strong` 还会生成全局 `AGENTS.override.md` 和 hooks.json 配置。

### 9.3 当前 Hook 职责

- `codex-session-start.js`：会话开始提醒 EKG 工作流
- `codex-user-prompt.js`：根据用户 prompt 注入经验与论文提示
- `codex-bash-guard.js`：阻止直接改写受管存储文件
- `pre-edit.js`：文件编辑前提醒相关经验，并附带相关论文
- `task-complete.js`：生成 capture candidate

## 10. Panel 架构

Panel 当前支持两种运行方式：

- 静态导出：输出 HTML
- 本地服务：启动 HTTP server，并支持简单交互动作

Panel ViewModel 主要包含：

- stats
- recent_experiences
- recent_papers
- top tags / techs / files
- top topics / venues
- graph_summary
- graph_view
- projects / active_project
- pending_candidates
- pipeline

## 11. 多 agent 并发控制

当前并发策略不是数据库事务外放，而是“CLI/runtime 层写锁 + backend 持久化”：

1. 写入命令通过 `withWriteLock` 或 `mutateRuntime` 串行化。
2. 后端写 SQLite 时使用事务。
3. 读取允许并发。

这保证了：

- 多 agent 能同时查
- 多 agent 不会直接并发写坏索引

## 12. 当前已知缺口

当前技术债和后续改造点主要有：

1. 中文设计文档长期滞后于实现，需持续同步。
2. stale 检测目前仍以文件 anchor 为主，语义级 stale 尚未做深。
3. 经验演化链还没有正式模型，例如 `supersedes`、`superseded_by`。
4. Prompt/Pre-edit 对论文的利用已经接入，但仍偏轻量，尚未形成完整研究工作流。
5. 尚未提供 MCP / query server。
6. 图分析仍偏展示和解释，缺少真正的聚类、热点分析和时序分析。

## 13. 推荐后续演进路线

推荐继续沿以下顺序演进：

1. 完善中文文档与 README/Guide 的同步机制。
2. 扩大自动 ingest 来源，但保持 review gate。
3. 补齐经验演化与失效治理模型。
4. 提供 MCP / query server。
5. 引入更系统的图分析与研究视图。
