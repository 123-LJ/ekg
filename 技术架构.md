# EKG 技术架构

## 1. 架构定位

EKG 是一个面向编码代理与工程协作的本地知识图谱系统。

它解决的不是“替代代码搜索”，而是把已经验证过的工程经验、论文知识、项目上下文和待审核候选，组织成可查询、可解释、可回放的本地记忆层。

当前架构目标有四个：

1. 在改代码前尽量先回忆已有经验，减少重复踩坑
2. 在任务结束后先进入 review candidate，而不是直接写成正式知识
3. 让经验与论文进入同一图谱，支持联查和解释
4. 以本地优先、可审计、可迁移为原则，不依赖中心化服务

## 2. 当前分层

### 2.1 交互层

主要入口包括：

- CLI：`node scripts/ekg.js ...`
- Codex Hooks
- Claude Hooks
- 本地 Panel：静态导出或本地 HTTP 服务

### 2.2 运行时层

核心由以下能力组成：

- `buildRuntime`
- `loadRuntime`
- `mutateRuntime`
- `saveRuntime`
- `saveState`
- `withWriteLock`

职责：

- 加载 config / index / state / storage backend
- 在写入前后刷新索引、统计与图谱
- 用写锁保证多代理环境下的串行更新

### 2.3 领域模型层

当前一等实体：

- `Experience`
- `Paper`
- `Project`
- `Capture Candidate`

辅助节点会在图中动态生成，例如：

- file
- concept
- tag
- tech
- topic
- keyword
- author
- venue
- canonical

### 2.4 检索与图谱层

核心能力包括：

- `queryExperiences`
- `queryPapers`
- `traceExperiences`
- `buildKnowledgeGraph`
- `explain`
- `path`
- `survey`

这一层既支持直接检索，也支持沿关系链和图路径做解释。

### 2.5 持久化层

持久化组合如下：

- 主存储：`ekg-out/ekg.sqlite`
- 兼容镜像：`ekg-out/ekg.json`
- 状态文件：`ekg-out/state.json`
- 报告输出：`ekg-out/reports/EKG_REPORT.md`
- 可读副本：`experiences/*.md`、`papers/*.md`

设计原则是：

- SQLite 负责结构化、增量、并发友好的主存储
- JSON/Markdown 负责审阅、diff、导出和备份友好性

### 2.6 宿主集成层

当前宿主集成主要负责：

- 会话开始提醒 EKG 工作流
- 用户提问时注入相关经验与论文提示
- 编辑前查询相关知识
- Bash guard 防止直接破坏受管存储
- 任务完成时创建 capture candidate

## 3. 数据流

### 3.1 查询流

典型流程：

1. 用户或 Hook 提供关键词、文件路径或任务描述
2. 运行时加载 `index + state + config`
3. 检索层先做规则命中，再叠加轻量语义评分
4. 如需解释，再补图谱邻接节点和路径信息
5. CLI / Hook / Panel 输出结果

### 3.2 写入流

典型流程：

1. 通过 CLI 或 Hook 发起写入
2. 获取 `.ekg.lock`
3. 修改运行时对象
4. 刷新索引、统计和图谱
5. 写入 SQLite
6. 导出 JSON / Markdown / Report 镜像

### 3.3 Capture / Review 流

典型流程：

1. `task-complete` 或 `ingest` 收到任务总结
2. 生成 capture candidate
3. 进入 `NEEDS_REVIEW / UNCERTAIN`
4. 人工执行 `capture-accept` 或 `capture-dismiss`
5. 接受后转为正式 Experience

当前补强点：

- candidate 被过滤时不再静默退出，而会返回 skip reason
- 长任务总结会先压成 compact text，再进入候选判断和展示
- 原始长文本仍保存在 origin 元数据里，避免信息损失

### 3.4 论文导入流

典型流程：

1. `paper-import` 调 OpenAlex 或 Semantic Scholar
2. 标准化为 `Paper` 模型
3. 自动补齐 aliases / canonical_terms / suggested_canonical_terms
4. 按 DOI / arXiv / title 去重或合并
5. 写入 SQLite、JSON 与 Markdown

## 4. 检索架构

### 4.1 Experience 检索

Experience 检索使用“规则 + 轻量语义”的混合模式。

规则信号包括：

- text token
- file
- concept
- tag
- tech

语义信号包括：

- alias expansion
- trigram overlap
- field coverage
- canonical concept overlap

同时，演化治理已进入检索层：

- 被 `SUPERSEDED` 的经验不会删除
- 但会在默认排序中降权
- 若存在更新版本，会给出 recommended current version 提示

### 4.2 Paper 检索

Paper 检索字段包括：

- title
- abstract
- summary
- findings
- limitations
- notes
- authors
- topics
- keywords
- venue / year
- canonical_terms

当前论文层采用“全量 + 紧凑摘要”的分级方式：

- `abstract` 保留更完整文本
- `summary` 保留较短、适合检索与面板展示的紧凑版本
- `paper_file` 继续保留可读副本

### 4.3 多语言 canonical retrieval

当前已支持：

- multilingual alias registry
- exact canonical hit
- suggested canonical term
- 中英混合概念命中

## 5. 演化治理架构

这是当前版本新增的重要结构。

目标不是覆盖旧经验，而是保留版本链并控制推荐顺序。

当前能力：

- 新经验可通过 `supersede <new> <old>` 标记取代关系
- 旧经验状态改为 `SUPERSEDED`
- 查询层对 superseded 结果降权
- `query / explain / panel` 可显示版本链上下文
- 报表和状态统计包含 superseded count

当前还没有完全做完的部分：

- 自动 stale 与 supersede 的联动
- 更细粒度的版本分支管理
- 多版本聚合视图

## 6. 长文本与论文存储策略

当前对长文本采用“压缩展示 + 分级存储”的混合方案。

### 6.1 为什么不能只做截断

如果只截断：

- capture 候选容易丢关键动作句
- 论文摘要容易只剩前半段背景，失去可检索性
- review 阶段会误判为信息不完整

### 6.2 当前方案

- 对 capture candidate 使用规则式 compact text
- 对 paper `summary` 使用 compact text
- 对原始长内容继续保留：
  - candidate origin 中保留 full task / summary
  - paper 中保留 full abstract
  - `paper_file` 保留 Markdown 副本

这个方案的好处是：

- 检索和 review 层不会被长文本淹没
- 原始信息仍然可追溯
- 不需要先引入更复杂的摘要服务才能稳定运行

## 7. Panel 架构

Panel 当前支持两种运行方式：

- 静态 HTML 导出
- 本地 HTTP server

当前可展示：

- stats
- recent experiences
- recent papers
- graph summary / graph view
- projects / active project
- pending capture candidates
- pipeline
- candidate review actions
- experience version-chain context

## 8. 当前已知缺口

1. 结构化外部接入还没有完成，缺少 MCP / query server
2. stale 分析仍以文件 anchor 为主，语义 stale 还弱
3. 研究工作流还不够强，缺论文分组、方向卡片、候选筛选
4. 图分析仍以解释和展示为主，还没有成熟聚类/热点分析
5. compact text 目前是规则式压缩，不是更高级的语义摘要

## 9. 后续建议路线

建议顺序：

1. 完成 MCP / query server
2. 增强 papers + survey 研究工作流
3. 做更强的 stale / invalidation / evolution 联动
4. 增加高级图分析能力
5. 视需要再引入更强的语义摘要或压缩能力
