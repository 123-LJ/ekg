# EKG 状态说明

## 当前阶段

**当前阶段标识：`phase-2-plus`**

可以用一句话概括：

EKG 已经不是早期的“经验笔记脚本”，而是一个以 SQLite 为主存储、同时支持经验层、论文层、项目上下文、capture/review、宿主 Hook 和本地 panel 的本地知识图谱工具。

## 阶段拆解

### Phase 1

已完成：

- Experience 基础模型
- CLI 查询与新增
- explain / path / report
- 基础 Hook

### Phase 2

大部分已完成：

- capture candidate 流程
- review / accept / dismiss
- SQLite 主存储
- JSON / Markdown 兼容镜像
- stale-check 原型
- 项目上下文
- 宿主安装脚本
- 写锁保护
- 便携备份导出与恢复

### Phase 3

已有预览能力：

- graph view
- trace
- local panel
- 论文层与 survey
- 多语言 canonical retrieval

仍未完成：

- MCP / query server
- 聚类分析
- 更成熟的研究工作流
- 演化链与版本治理

## 当前真实能力边界

已经稳定可用的部分：

1. 基本经验工作流已经闭环。
2. capture/review 已经能防止自动写入污染正式经验库。
3. SQLite + JSON 镜像 + Markdown 副本的组合已经稳定。
4. 项目上下文、Panel、备份、宿主 Hook 都已可用。
5. 论文与经验已进入同一图谱，可做 `survey` 和联查。

仍然只是原型或半成品的部分：

1. stale 检测目前仍以文件 anchor 为主。
2. ingest 仍偏保守，重点是“少噪声”，不是“全自动覆盖”。
3. 论文层虽然可导入可查询，但还没有形成更强的研究工作流编排。
4. 经验演化链还没有正式模型。

## 当前运行建议

建议的日常使用顺序：

1. 先用 `project-use` 或 `project-register` 进入当前项目上下文。
2. 改代码前先 `query`，必要时看 `trace`。
3. 做新方向前先 `paper-query` 或 `survey`。
4. 任务结束后用 capture candidate 流程，而不是直接相信自动抽取。
5. 周期性执行 `report`、`panel`、`backup-export`。

## 建议优先级

接下来最值得做的不是继续堆新入口，而是把现有能力打磨成真正稳定的工作流。

建议优先级如下：

1. **经验演化治理**
   需要引入 `supersedes`、`superseded_by`、`SUPERSEDED` 之类的正式模型，解决“旧方案仍被命中”的问题。

2. **更强的 stale 检测**
   当前更多是文件级漂移，后续需要概念级、模块级和路径级 stale 提示。

3. **研究工作流增强**
   现在已有 papers + survey，但还缺论文分组、方向卡片、候选论文筛选等能力。

4. **MCP / query server**
   这会让外部 agent 能以结构化方式读取 EKG，而不是只依赖 CLI 和 Hook。

5. **高级图分析**
   包括热点、聚类、路径解释增强、时序统计等。
