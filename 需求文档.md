## 1. 项目概述

| 字段 | 内容 |
|------|------|
| 项目名称 | EKG（Experience Knowledge Graph） |
| 项目目标 | 为 AI 编码协作提供一个“经验索引 + 经验图谱”系统，减少重复探索、重复犯错和上下文浪费 |
| 目标用户 | 使用 Claude Code / Codex / 多 Agent 协作的开发者 |
| 成功标准 | AI 在写代码前能命中相关经验；经验可持续累积；查询可解释；不会明显打扰编码节奏 |

| 材料类型 | 内容 | 用途 |

## 2. 范围定义

### 2.1 范围内

- 经验记录的目录结构定义
- 经验节点与关系模型定义
- `ekg.json` / `EKG_REPORT.md` 等核心产物定义
- `/ekg` 命令族需求定义
- PreToolUse Hook 的分级注入策略
- 冷启动、置信度、失效状态等机制定义
- Phase 1 / 2 / 3 的实现边界拆分

### 2.2 暂不包含

- 完整 MCP Server 实现
- 向量检索与 embedding 存储
- 图形化 Web UI
- 自动解析所有历史对话
- 跨团队共享的远程图数据库
- 完整的 Graph HTML 可视化实现

## 3. 功能需求

### 3.1 P0 核心功能

| 功能ID | 功能名称 | 验收标准 |
|--------|----------|----------|
| F01 | 手动经验录入 | 支持通过 `/ekg add <描述>` 或文件方式新增经验，经验可落盘且可再次读取 |
| F02 | 结构化经验存储 | 每条经验至少包含问题、解法、标签、置信度、适用范围、来源、锚点 |
| F03 | 基础经验查询 | 支持按关键词、标签、文件路径或概念检索经验，并返回可解释命中理由 |
| F04 | 分级注入 | 编码前 Hook 可按 Level 0/1/2 决定静默、摘要注入或详细注入 |
| F05 | 状态管理 | 经验支持 `ACTIVE / NEEDS_REVIEW / STALE / ARCHIVED` 状态 |
| F06 | 冷启动种子 | 系统可在空库时加载一组 L3 原则级种子经验，避免完全无结果 |

### 3.2 P1 重要功能

| 功能ID | 功能名称 | 验收标准 |
|--------|----------|----------|
| F11 | 置信度标签 | 每条经验必须有 `CONFIRMED / INFERRED / UNCERTAIN` 之一 |
| F12 | 语义锚定 | 经验支持按文件、组件、模块、commit hash 等语义锚点绑定 |
| F13 | 增量更新 | 新经验写入时不重建全部图谱，支持状态文件记录上次处理位置 |
| F14 | 周期性报告 | 生成 `EKG_REPORT.md`，汇总热点问题、核心概念、待审经验和失效经验 |

### 3.3 P2 后续能力

| 功能ID | 功能名称 | 验收标准 |
|--------|----------|----------|
| F21 | 自动提取 | 从 git log、commit message、对话片段中自动生成候选经验 |
| F22 | 聚类分析 | 当经验数量达到阈值后，支持按领域或技术栈做社区发现 |
| F23 | MCP 查询 | 提供结构化查询能力给外部 AI 工具调用 |
| F24 | 交互式可视化 | 输出 `ekg.html` 或其他图谱可视化结果 |

## 6. 命令关系与触发

| 命令 | 触发动作 | 结果 | 前置条件 | 备注 |
|------|----------|------|----------|------|
| `/ekg` | 构建或更新图谱 | 刷新索引与报告 | 目录存在 | Phase 1 可做轻量更新 |
| `/ekg add <描述>` | 新增经验 | 生成经验条目 | 输入非空 | 默认写入 L1 |
| `/ekg query <关键词>` | 查询经验 | 返回候选经验及命中原因 | 至少已有种子或经验 | 默认摘要输出 |
| `/ekg explain <概念>` | 解释概念经验 | 汇总概念相关经验 | 概念存在 | 偏阅读场景 |
| `/ekg path <A> <B>` | 查找经验路径 | 返回节点间关联路径 | 图中存在节点 | Phase 2 更有价值 |
| `/ekg review` | 审核经验 | 把候选经验转为确认经验 | 有 `UNCERTAIN` 条目 | 支持批量审阅 |

## 7. 页面状态与反馈（命令态）

| 模块 | 场景 | 表现 | 用户可执行操作 | 备注 |
|------|------|------|----------------|------|
| `/ekg add` | 成功 | 返回经验 ID 与摘要 | 可继续补锚点 | 不应静默成功 |
| `/ekg query` | 无结果 | 返回“无匹配经验” | 提示改关键词或新增经验 | 不算错误 |
| `/ekg review` | 无待审条目 | 返回空状态 | 无 | 不应失败 |
| Hook | Level 0 | 静默 | 无 | 不打扰 |
| Hook | Level 1 | 一句话摘要 | 可进一步查询 | 默认模式 |
| Hook | Level 2 | 完整经验摘要 | 可继续展开 | 高置信度或主动查询 |

## 8. 关键数据流转

### 8.1 关键数据流转

| 场景 | 输入数据 | 处理逻辑 | 输出结果 | 落库/缓存/同步 |
|------|----------|----------|----------|----------------|
| 手动新增经验 | 文本描述 + 可选标签 | 结构化整理为经验条目 | 新经验节点 | 落盘至经验文件与 `ekg.json` |
| 写代码前注入 | 当前文件路径 / 编辑目标 | 匹配相关经验与注入等级 | Level 0/1/2 注入结果 | 仅缓存命中记录 |
| 经验查询 | 关键词/标签/概念 | 检索经验并排序 | 候选经验列表 | 可记录查询日志 |
| 周期性分析 | 当前全部经验 | 统计热点、失效、待审状态 | `EKG_REPORT.md` | 落盘 |

### 8.2 关键业务流程

```text
用户/AI 发现一个有价值的坑
  -> 通过 /ekg add 记录经验
  -> 系统生成 Experience 节点
  -> 经验写入 experiences/ 与 ekg.json
  -> 后续编辑相关文件时 Hook 命中
  -> 系统注入摘要提醒
  -> 用户或 AI 可继续 /ekg query /ekg explain
  -> 当经验长期未验证或锚点变化过大时，转为 NEEDS_REVIEW / STALE
```

## 10. 业务规则

1. 经验默认从 L1 开始，不直接自动升级为 L2/L3。
2. `CONFIRMED` 经验优先于 `INFERRED` 和 `UNCERTAIN` 经验被注入。
3. Hook 默认使用 Level 1 摘要注入，不默认全量展开。
4. 行号不是可靠锚点，首版不作为主锚定方式。
5. 当锚定文件发生大幅变化时，经验至少要转为 `NEEDS_REVIEW`。
6. 无命中经验时不得制造“伪命中”。
7. 冷启动阶段必须预置少量 L3 原则级经验。

## 12. 验收标准

1. 至少能手动新增、查询、审阅经验。
2. 经验条目可落盘，并能在后续命令中再次检索到。
3. 查询结果必须说明命中原因，不允许只返回“相似经验”而不给依据。
4. Hook 必须支持 Level 0/1/2 三种行为。
5. 冷启动状态下系统不应完全空白，至少有一组 L3 种子经验。
6. 文档化后的目录结构可被整体复制到新环境继续使用。
