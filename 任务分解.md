## 6. 当前风险与处理

| 风险项 | 影响 | 当前处理策略 |
|--------|------|--------------|
| 阶段标识落后于代码能力 | 用户误判项目还停在 Phase 1 | 文档和 pipeline 文案统一改成 Phase 2+ |
| 自动抽取噪声污染经验库 | 错误经验会误导 AI | 继续坚持候选 -> 审核 -> 确认写入 |
| Hook 过度打扰 | 降低使用体验 | 只注入命中摘要，强匹配再展开 |
| 多 agent 并发写入 | 数据冲突或覆盖 | 使用 runtime/CLI 写入和 lock，禁止直接改 store 文件 |
| CDN 面板依赖 | 离线时图谱面板体验下降 | 后续可加 vendor fallback |
| 经验过期 | 旧方案误导新开发 | 后续补 stale 自动检测和演化记录 |

## 7. 下一步推荐

优先级从高到低：

1. **补齐 Phase 2 缺口**：强化自动 ingest、补完 stale 检测、经验演化/替代关系。
2. **强化 Phase 3 图谱能力**：cluster、热点分析、GraphML/GEXF 导出。
3. **完善面板**：本地 vendor fallback、候选审核入口、图谱筛选和搜索体验。
4. **服务化接入**：MCP/query server，让 Codex、Claude 和其他 agent 能结构化查询。
5. **持续验证**：每次真实需求后用 capture candidate 记录经验，确认后再 accept。

## 8. 待开发优先级清单（新增）

### TODO-P2-001：自动 ingest 候选经验生成

**优先级：P0**

目标：从 commit、task、对话摘要中自动生成 `capture candidate`，但不直接写入正式经验库。当前已落地首版 `ingest` 命令骨架。

范围：

- 从 `git log` / commit message 中识别 `fix:`、`bug:`、`hotfix:`、`feat:`、`refactor:` 等信号。
- 从 `hooks/task-complete.js` 或任务完成摘要中提取候选经验。
- 后续预留对话摘要输入源，但第一版不做全量聊天记录扫描。
- 自动生成的内容默认进入 `NEEDS_REVIEW + UNCERTAIN`。
- 必须经过 `capture-status` / `capture-accept --confirm` 后才能成为正式经验。

验收标准：

- 能执行类似 `node scripts/ekg.js ingest --since <commit>` 或等价命令生成候选经验。当前首版已支持 `ingest` 命令。
- 自动 ingest 不允许直接污染正式经验库。
- 有 commands 层和 CLI 集成测试。
- 文档说明触发规则、跳过规则和审核流程。

### TODO-P2-002：stale / NEEDS_REVIEW 自动检测

**优先级：P0**

目标：当经验锚定的文件发生明显变化时，自动提示经验可能过期，并把相关经验标记为 `NEEDS_REVIEW`。当前已落地首版 `stale-check` 骨架。

范围：

- 为经验 anchors 记录文件 hash、mtime 或 git commit 版本。
- 当锚定文件被删除、大幅重写或关键路径变化时，标记经验为 `NEEDS_REVIEW`。
- 第一版只做文件级检测，不做复杂行号级追踪。
- 后续可升级到语义锚点：函数名、组件名、模块名。

验收标准：

- 能执行类似 `node scripts/ekg.js stale-check` 或等价命令扫描经验 anchors。当前首版已支持基线建立和扫描。
- 文件删除/变化能让相关经验进入 `NEEDS_REVIEW`。
- `query` 和 panel 能显示 stale/review 状态。
- 有 storage、commands、CLI 测试覆盖。

### 备注

这两个 TODO 是 Phase 2 剩余缺口里收益最高的两项：

1. 自动 ingest 负责“自动发现可沉淀经验”。
2. stale 检测负责“防止旧经验误导后续 agent”。

两者都必须坚持：**先生成候选或 review 状态，不能绕过人工确认直接写入高置信经验。**
